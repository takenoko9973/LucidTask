# 基本設計書: デスクトップ常駐型タスク管理ウィジェット

## 1. システム要件と設計思想

- **対象OS** : Windows
- **アプリケーション形態** : デスクトップ常駐型透過ウィジェット
- **設計思想** : ワーキングメモリの限界を考慮し、デフォルトの表示タスクを最大4件に制限する。システムによる強制ソートと段階的開示により、ユーザーの認知負荷と判断コストを最小化する。

## 2. アーキテクチャ構成

- **バックエンド** : Rust (Tauriフレームワーク)
- **フロントエンド** : HTML, CSS, TypeScript
- **通信方式** : Tauri IPCによる非同期メッセージング

## 3. データモデルと永続化

### 3.1 データ構造

無期限タスクの廃止に伴い、`TaskType`列挙型を再定義する。

```rust
use serde::{Deserialize, Serialize};
use chrono::{DateTime, Local};

#[derive(Serialize, Deserialize, Clone)]
pub enum TaskType {
    Deadline(DateTime<Local>), // 期限付きタスク
    Daily,                     // デイリータスク
}

#[derive(Serialize, Deserialize, Clone)]
pub struct Task {
    pub id: String,
    pub title: String,
    pub task_type: TaskType,
    pub is_pinned: bool,       // 固定化フラグ (最大3件の制約は追加ロジックで担保)
    #[serde(skip)]
    pub completed_at: Option<DateTime<Local>>, // 完了日時 (JSONには保存しない)
}
```

### 3.2 データのライフサイクル

- **保存先** : ローカルJSONファイル。
- **未完了タスク** : 作成・更新時にメモリとJSONファイルに同期。
- **削除済みタスク** : メモリおよびJSONから即時破棄。
- **完了済みタスク** : 完了操作時に`completed_at`を記録し、JSONの保存対象外とする。メモリ上には保持し、完了から3日 (72時間) 経過後に定期処理で破棄。

## 4. UI/UX設計と表示ロジック

### 4.1 メインウィンドウの表示・ソートロジック

手動の並び替えを排除し、以下の優先順位でシステムがソートを強制する。

1. 固定化タスク (最大3件)
2. 期限超過・当日の期限タスク
3. デイリータスク
4. 明日以降の期限タスク (期限が近い順)

### 4.2 段階的開示とスクロール制御

- **初期状態** : 上位4件のタスクのみを描画し、ウィンドウ高さを4件分に固定する。最下部に展開ボタンを配置する。
- **展開状態 (タスク数5〜9件)** : 展開ボタン押下時、タスクの数に合わせてウィンドウの縦幅を動的に拡張し、全件を表示する。
- **制限状態 (タスク数10件以上)** : ウィンドウの縦幅拡張の上限を「タスク9件分の高さ」または「指定した最大ピクセル数」で停止させ、オーバーフロー領域に対して垂直スクロールバーを表示する。

### 4.3 タスクカードのUI定義

- **タスク名の切り詰め** : CSSプロパティ `white-space: nowrap; overflow: hidden; text-overflow: ellipsis;` を適用し、1行で強制的に切り詰める。
- **ツールチップ** : タスク名にマウスホバーした際、HTML標準の`title`属性、またはカスタムツールチップ実装により全文を表示する。
- **視覚的タグ付け** : タスクの種類や状態 (期限超過、デイリーなど) を直感的に判別できるよう、カードの端にアイコンや色付きのインジケーターを配置する。

## 5. サブウィンドウとシステム連携

- **タスク操作ダイアログ** : 新規登録・編集用の別ウィンドウをIPC通信を用いて生成。
- **システムトレイ** : タスクトレイにアイコンを常駐し、各種メニューを提供。
- **スタートアップ登録** : OSのレジストリへ自動起動設定を登録。
